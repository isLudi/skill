# 新人过程转化 `x_qi_count` 重复与 `pingyou_jg` 上传故障

## 1. 适用范围

用于排查市场顾问部【新人】前期过程转化数据看板、数据中心模型 `2688`（`新人过程转化数据`）以及 `pingyou_jg.xlsx` / `temp_table.dingxi01_pingyou_jg` 的以下问题：

- 同一顾问在多个期次重复出现相同 `x_qi_count`。
- `x_qi_count` 出现 `3,2` 等非法值。
- 本地 Excel 正常，但上传后 `qici='1'`、字段错位或重复键增加。
- 修复临时表后，模型仍没有最新暑期期次。

## 2. 结论：先区分四层故障

| 层级 | 本次直接原因 | 典型症状 | 正确修复点 |
|---|---|---|---|
| Excel 业务维护 | `x_qi_count` 由人工延续，未按顾问有效期次重排；另有 1 条完全重复行和 `3,2` 非法文本 | 同一顾问重复 `1`—`4` | 修复 `pingyou_jg.xlsx`，不是在模型结果外层去重 |
| Excel → 临时表上传 | 多 sheet 导入选错数据页；14 列中存在空值时，本次平台解析出现列错位 | 回执仅 63 行，或上传后 `min(qici)='1'` | 只保留 `Sheet4`、补全 14 列、全量 overwrite，并核对回执 |
| 模型关联 | 模型 `2688` 直接透传 `ppg.x_qi_count`，不会计算或纠错 | 源表错，模型同步错；但同一期次顾问仍可能唯一 | 用关联基数证明是否 join 放大；不要凭“重复”二字直接加 `distinct` |
| 暑期日历 | 旧 SQL 对线索期次、开课期次和员工入部门期次回退到自然周周五 | 临时表已有 `20260716期`，模型最大期次仍停在 `20260710期` | 为所有日期角色补相同连续业务日历，再保留历史回退 |

## 3. 为什么确认不是 SQL join 放大

模型 `2688` 的最终层使用：

```sql
left join temp_table.dingxi01_pingyou_jg ppg
  on ppg.employee_email_name = zz.employee_email_name
 and ppg.qici = zz.period_name
where ppg.x_qi_count not in ('9')
```

`x_qi_count` 没有在 SQL 中计算，也没有 `row_number` 重排；它是 Excel 源字段的直接透传。

修复前的完整模型结果有 360 行，`period_name + employee_email_name` 重复键为 0，但 `employee_email_name + x_qi_count` 在有效值 `1`—`4` 中有 29 个重复组。也就是说，同一期次顾问粒度没有被 join 放大，重复的是同一顾问跨期的人工序号。

源 Excel 同时发现：

- 数据行 5921 行，其中 1 条完全重复行。
- 37 个顾问有效 `x_qi_count` 重复组。
- 1 个非法值 `3,2`。

这些证据把根因定位到手工表维护，而不是事实表聚合或最终 join。若未来 `period_name + employee_email_name` 也出现重复，再单独检查临时表键和事实聚合粒度。

## 4. Excel 维护故障与业务规则

`x_qi_count` 是顾问新人观察口径的有效承接期次序号，不是累计次数或可求和指标：`1`—`4` 表示前四个有效期次，`9` 表示不进入新人前四期展示。

修复规则固定为：保留原有 `9`；只对原来非 `9` 的行按同一顾问 `qici` 升序编号 `1,2,3,4`，第 5 个及以后改为 `9`。公式、示例和终版 14 字段约束见 `../tables/temp_table.dingxi01_pingyou_jg.md`。

本次按该规则删除 1 条完全重复行并改正 77 个 `x_qi_count` 单元格，终版 5920 行；有效序号重复组和非法值均降为 0。

## 5. 两类上传陷阱

### 5.1 多 sheet 导入非预期数据页

候选文件一度额外包含 `Sheet1`，其中只有 63 行数据。首次导入回执也只有 63 行，证明上传链路没有使用预期的完整 `Sheet4`。

处理原则：上传文件只能保留 `Sheet4`；不要假设平台一定按活动 sheet、名称或最大行数自动选择正确数据页。导入回执行数必须与本地 `Sheet4` 数据行数完全一致。

### 5.2 空单元格导致列错位

第一次上传 5920 行版本后，线上回读出现：`min(qici)='1'`、29 个期次、2 个 `employee_email_name + qici` 重复键。异常样本集中在本地 `grade` 为空的 6 行，平台回读中这些行发生字段整体错位。

本次用可追溯证据补为 `高中`：优先取同一顾问、同一渠道最近期次；无该证据时取同一期次、同部门、同渠道众数。补齐后再次 overwrite，线上回读恢复为 `20260101期` 至 `20260716期`、28 个期次、重复键 0。

因此，`pingyou_jg` 上传前必须保证 14 列全非空。不要依赖平台替空值保留列位置，也不要用空字符串作为“未知”长期上传；无法确认业务值时应先停在本地待人工补全。

另外，4 个 `Liyeyang` 邮箱前缀已统一为小写 `liyeyang`。`曹可悦` 跨期对应过不同经理/小组属于历史组织变化，不能只因一人多主管就删除历史行。

## 6. 为什么修完 Excel 仍缺 `20260716期`

只修临时表后，完整模型从 360 行增至 408 行，有效 `x_qi_count` 重复组已为 0，但最大期次仍是 `20260710期`。原因是模型对以下日期角色仍使用自然周回退：

- 线索 `group_period` → `period_name`。
- 行课 `begin_time` → `class_period`。
- 员工进入市场顾问部 `begin_time` → 入部门期次。

本次为 `lead_period` 和 `class_period` 统一补入连续暑期日历：

| 期次 | 日期范围 |
|---|---|
| `20260716期` | `2026-07-14` 至 `2026-07-19` |
| `20260722期` | `2026-07-20` 至 `2026-07-25` |
| `20260728期` | `2026-07-26` 至 `2026-07-31` |
| `20260803期` | `2026-08-01` 至 `2026-08-06` |
| `20260809期` | `2026-08-07` 至 `2026-08-12` |

未命中上述区间时才回退历史自然周逻辑。替换 SQL 验证结果为 422 行、17 个期次、最大期次 `20260716期`；`period_name + employee_email_name` 重复键 0，顾问有效 `x_qi_count` 重复组 0。

## 7. 标准修复与上线顺序

1. 先在原始 Excel 统计：完全重复行、`employee_email_name + qici` 重复键、顾问有效 `x_qi_count` 重复组、非法值和必填空值。
2. 按稳定业务规则修复 `x_qi_count`，粘贴为值，删除临时辅助列；终版不新增字段。
3. 只保留 `Sheet4`，确认 14 个表头、5920 行、无公式、无空值、邮箱前缀小写。
4. 上传到登记表 `dingxi01_pingyou_jg`，使用 `overwrite`；回执行数必须等于本地数据行数。
5. 在线回读行数、期次范围、非法值、两个唯一性检查；任一不符都不得刷新数据中心。
6. 用模型当前完整 SQL只读验证结果粒度。若最新期次仍缺失，逐一核对 lead/class/org 日期角色的业务日历。
7. 数据中心替换必须走精确 Hash 计划、完整替换、预览、保存回读和新 `SUCCESS`；随后同步当前 canonical SQL 回知识库。
