# Validate Data Skill 中文使用说明

## 这个 skill 是做什么的

`validate-data` skill 用于在分析结果交付前做数据校验和方法审阅。它适合检查 SQL 逻辑、指标口径、聚合粒度、分母选择、join 是否膨胀、时间周期是否可比、图表是否误导，以及结论是否真的被数据支持。

这个文件是中文上手说明，实际执行规则仍以同目录下的 `SKILL.md` 为准。

## 适合使用的场景

- 向业务、管理层或项目相关方汇报前，审阅一份分析报告。
- 检查 SQL 查询结果是否合理，尤其是转化率、留存率、GMV、收入、排名、占比等指标。
- 发现看板数值异常时，系统排查是否存在口径、过滤、分区、join 或聚合问题。
- 对图表和结论做交付前 QA，判断是否存在误导性表达。
- 比较月环比、同比、季度、期次等周期时，检查是否存在不完整周期或分母变化。
- 复核 Excel、Markdown、Notebook、SQL 结果集或看板文档中的分析逻辑。

## 基本使用方法

在 Codex 中直接说明要使用 `validate-data` skill，并提供需要审阅的分析内容即可。分析内容可以是 SQL、查询结果、指标解释、图表描述、Markdown 报告、Excel 摘要，或一段准备发给业务方的结论。

推荐提供这些信息：

- 分析要回答的问题。
- 使用的数据源、表名或文件路径。
- SQL 查询和核心计算逻辑。
- 关键指标定义和输出粒度。
- 查询结果样例或汇总结果。
- 计划对外表达的结论。
- 已知限制、假设或待确认事项。

## 常见工作流

### 方法论检查

先确认分析问题、数据源、样本范围、指标定义和比较基准是否合理。重点问清楚“分析对象是谁”“谁被排除在外”“时间周期是否完整”“口径是否和业务理解一致”。

### 计算逻辑检查

重点检查：

- `group by` 粒度是否与最终输出粒度一致。
- rate、ratio、conversion 等指标的分母是否稳定且非零。
- `count(*)` 是否应该改为 `count(distinct ...)`。
- `left join`、`inner join`、多对多 join 是否改变了数据行数。
- 明细求和是否能对上汇总结果。

### 合理性检查

检查数值是否在合理范围内，例如百分比是否在 0 到 100 之间，环比变化是否过大，结果是否出现完全相同的重复值、整齐的整数、异常的 0% 或 100%。

### 表达和交付检查

检查图表标题、坐标轴、时间范围、样本量、限制说明和结论措辞。不要把相关性表述成因果关系，不要忽略关键 caveat。

## 和本地 SQL skill 的配合方式

`validate-data` 适合作为 `sql-query-writer-for-dashboard` 的后置校验工具：

- 先用 `sql-query-writer-for-dashboard` 生成或修复 Presto SQL。
- 再用 USQL RestAPI 或查询平台拿到结果。
- 最后用 `validate-data` 检查结果、口径、结论和交付风险。

如果任务是“生成 SQL”，优先使用 `sql-query-writer-for-dashboard`；如果任务是“检查这份分析是否可靠”，优先使用 `validate-data`。

## 使用注意

- 该 skill 不替代真实业务口径确认；发现口径不明确时应标注为待确认。
- 对高风险汇报，最好同时提供 SQL、结果样例和结论文本，否则只能做有限审阅。
- 如果发现问题，应先修正分析再重新校验。
- 对周期对比、排名、转化率、留存率、收入类指标，要特别关注分母、去重、时间窗口和 join 膨胀。
- 该 skill 源自 Anthropic 的 `knowledge-work-plugins/data`，已做 Codex 本地化使用说明；后续如继续改造，应避免引入上游 slash-command 或外部连接器占位表述。
