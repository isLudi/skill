# Code Simplifier Skill 中文使用说明

## 这个 skill 是做什么的

`code-simplifier` skill 用于在不改变功能和行为的前提下，简化并优化代码的清晰度、一致性和可维护性。它更关注最近修改过的代码，适合在完成一次功能开发、修复 bug、重构局部逻辑或整理代码风格之后使用。

这个 skill 来源于 Anthropic 的 `claude-plugins-official/plugins/code-simplifier`，本地已保留原始 Claude plugin 结构，并额外提供 `SKILL.md` 供 Codex 识别。这个文件是中文上手说明，实际执行规则仍以同目录下的 `SKILL.md`、`agents/code-simplifier.md`、`.claude-plugin/plugin.json` 和 `LICENSE` 为准。

## 适合使用的场景

- 简化最近新增或修改的函数、组件、脚本、SQL 构造代码或业务逻辑。
- 降低不必要的嵌套、重复分支、冗余变量和过度抽象。
- 把晦涩的变量名、函数名或中间状态改得更直观。
- 调整代码结构，使其更符合项目已有风格和约定。
- 清理描述显而易见逻辑的注释，保留真正有解释价值的注释。
- 将嵌套三元表达式改成更清晰的 `if/else` 或 `switch`。
- 在提交前做一次局部代码可读性和可维护性整理。

## 不适合使用的场景

- 需要改变业务行为、功能输出、接口契约或数据结构语义的任务。
- 需要大范围架构改造、模块拆分或跨系统迁移的任务。
- 只追求代码行数更少，而不关心可读性和可调试性的任务。
- 还没有测试、样例或上下文可以验证行为是否保持一致的高风险改动。
- 用户明确要求“不要重构”“不要动风格”“只修这个 bug”的场景。

## 基本使用方法

在 Codex 中直接描述需要简化的代码范围即可。请求中包含“简化代码”“整理最近改动”“提升可读性”“保持功能不变地重构”“去掉重复逻辑”等上下文时，Codex 可以根据 skill 描述决定是否加载 `code-simplifier`。

推荐提供这些信息：

- 需要处理的文件、目录或最近改动范围。
- 哪些行为、输出、接口或测试必须保持不变。
- 项目已有的代码风格约定，例如命名、导入顺序、组件写法、错误处理方式。
- 是否允许修改测试、文档或调用方。
- 是否需要运行测试、类型检查、lint 或快照对比。
- 哪些代码只是临时实验，哪些代码必须稳定保留。

## 常用 Prompt 模板

### 简化最近修改的代码

```text
请使用 code-simplifier skill 检查当前工作区最近修改的代码。
要求只做可读性和可维护性整理，不改变任何功能、输出、接口或数据语义。
完成后请列出修改点，并说明你运行了哪些验证命令。
```

### 简化指定文件

```text
请使用 code-simplifier skill 简化 <文件路径> 中的实现。
重点处理重复逻辑、过深嵌套、命名不清晰和不必要的抽象。
不要改变外部接口和现有行为；如发现可能影响行为的改动，请先说明风险。
```

### 整理 React 组件

```text
请使用 code-simplifier skill 整理 <组件文件路径>。
保持 UI 行为和 props 契约不变，优化组件结构、变量命名、条件渲染和辅助函数拆分。
避免为了减少行数而写出难读的嵌套三元表达式。
```

### 提交前代码清理

```text
请使用 code-simplifier skill 对本次 git diff 做提交前清理。
只处理当前 diff 涉及的代码；不要扩大到无关文件。
请保持行为不变，并在完成后给出 git diff 摘要和验证结果。
```

### 保持行为不变的局部重构

```text
请使用 code-simplifier skill 重构 <文件路径> 中的 <函数名/模块名>。
目标是让逻辑更直观、更容易维护。
必须保持输入输出、异常行为、边界条件和性能特征基本不变。
```

## 常见工作流程

### 1. 先确认范围

优先处理最近修改或用户明确指定的文件。除非用户要求全仓库整理，不要主动扩大范围。

### 2. 理解现有行为

先阅读调用方、测试、类型定义和关键边界条件。对于业务代码，先确认哪些输出、状态变化或副作用不能改变。

### 3. 选择低风险改动

优先做局部、可验证的清理，例如：

- 合并重复分支。
- 提取清晰命名的中间变量。
- 拆分过长函数中的局部辅助逻辑。
- 删除无意义注释。
- 把复杂条件表达式改成直观的分支结构。

### 4. 保持项目风格

遵循项目已有约定，而不是引入新的个人偏好。若项目中有 `CLAUDE.md`、`AGENTS.md`、`CONTRIBUTING.md`、lint 配置或格式化配置，应优先参考这些文件。

### 5. 验证行为

根据项目类型运行合适的验证命令，例如：

```bash
npm test
npm run lint
npm run typecheck
pytest
go test ./...
cargo test
```

如果无法运行测试，应明确说明原因，并指出剩余风险。

## 依赖和环境

`code-simplifier` 本身不依赖额外脚本或运行时工具。它主要依靠 Codex 阅读和编辑代码。

实际验证时可能需要项目自身的工具链，例如：

- JavaScript / TypeScript 项目的 `npm`、`pnpm`、`yarn`、`tsc`、ESLint。
- Python 项目的 `pytest`、`ruff`、`mypy`。
- Go 项目的 `go test`、`gofmt`。
- Rust 项目的 `cargo test`、`cargo fmt`、`cargo clippy`。
- 项目自带的格式化、测试或 CI 检查命令。

## 使用注意

- 核心原则是保持行为不变；任何可能改变行为的调整都应先说明风险。
- 不要为了“更短”牺牲可读性，避免过度压缩、复杂链式调用和晦涩的一行式写法。
- 不要删除有业务解释价值、警示价值或外部约束说明的注释。
- 不要把多个职责强行合并到一个函数或组件里。
- 不要在没有上下文的情况下重命名公共 API、导出符号、数据库字段或配置键。
- 修改后应尽量运行测试、类型检查或 lint；无法验证时要说明。
