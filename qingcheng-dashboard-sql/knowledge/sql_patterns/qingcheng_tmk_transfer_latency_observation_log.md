# 青橙 TMK 转正常线索链路产出与延迟观测台账

## 1. 用途与解释边界

本台账持续记录青橙 TMK 潜客转正常线索链路在不同底表中的实际到数时间、字段覆盖率和查询证据，供后续向业务和数据开发组说明取数延迟卡点。

- 当前记录基于 `2026-07-17` 单批业务操作，只能作为实际观测，不是数据表正式 SLA。
- 必须区分三种状态：物理分区存在、目标线索进入该分区、目标字段可回补。物理分区有数据不代表目标批次已经产出。
- `data_lake_fuwu.dwd_crm_leads_rt` 没有 `dt/hour` 分区，无法仅凭查询结果反推出精确入库时刻；只能记录首次查询到该关系的时间上界。
- 业财表未命中目标线索时，可能是尚未成交，也可能是订单归因尚未产出。没有业财行时不得把金额 `0` 直接解释为已确认未成交。

## 3. 当前可行链路

稳定主链应从 CRM 模型关系和私海分配事实出发，不得先用潜客小时宽表限制左侧集合：

1. `data_lake_fuwu.dwd_crm_leads_rt` 取 `model_type=1` 的青橙 TMK/规划系统潜客。
2. 同表用正常线索 `model_type=0` 的 `previous_model_id = 潜客 crm_leads_id` 找到转移后的正常线索 ID。
3. `service_dw.dwd_crm_assign_private_detail_hf.lead_id = 转移后的正常线索 ID`，按 `assign_time, private_sea_id` 取首次承接顾问；需要转手历史时保留全部私海记录。
4. 左连接 `bdg_ba.app_crm_prelead_cost_gmv_full_link_data_hf`，按潜客 ID 回补 TMK 顾问、期次、渠道和潜客过程字段。
5. 左连接 `service_dw.dm_crm_lead_stats_detail_hf`，按正常线索 ID 回补当前截面顾问和截面期次。
6. 左连接 `service_dw.dws_crm_order_lead_attribute_income_refund_stats_detail_hf`，按正常线索 ID 回补成交、成交年级、科目、主讲、收入、退款和净收入。

本批反向从私海分配表定位后，可在 `dwd_crm_leads_rt` 找到全部 17 条 `正常线索 -> previous_model_id -> 潜客` 关系。此前正向查询只有 2 条，是因为先从潜客小时宽表构造左侧集合，15 条尚未进入该表的潜客被提前过滤，并非 CRM 模型关系只有 2 条。

## 7. 当前卡点结论

1. 转移主链已经跑通：`dwd_crm_leads_rt` 可取 17 条正常线索及潜客关系，私海表可取 17 条首次承接，因此用户 ID、转移线索 ID、年级、线索数量、承接顾问已经可稳定输出。
2. 潜客小时宽表的 15 条缺失已在跨日后补齐。`dt=20260718/hour=04/06/08/10/12` 均命中 17/17，TMK 顾问、渠道、期次、年级和有效线索字段现已完整；该表仍存在明显跨日到数现象，但本批不再是字段覆盖瓶颈。
3. 业务页面规则与数据字段仍不一致：17 条目标记录在潜客宽表中的 `rule_name` 全部是 `0716期-青橙公海-SEC未加好友-高二/高三`，任意 `SEC后转` 规则仍为 0。当前只能通过 DWD 转移关系和私海分配事实识别“已转移”，不能用 `rule_name like '%SEC后转%'` 识别本批转移。
4. 截至 `20260718/hour=12`，当前线索截面表仍未包含 17 条新线索，无法从截面验证当前承接顾问和截面期次；这不影响使用私海历史表取首次承接，但若业务要查询“当前顾问”，仍需等待或修复截面同步。
5. 截至 `20260718/hour=12`，业财合法青橙范围内仍未命中目标线索，但范围内分区总量持续增长，已排除整表停更。未命中既可能是尚无成交，也可能是订单归因尚未产出；当前已知另一张财务明细只有用户级关联、没有确认的转移线索 ID 键，不能用它直接证明本批线索成交。金额空值不得解释为 0，成交状态也不得直接解释为“否”。
6. 查询实现仍必须以 DWD 转移关系或私海分配集合为主表，再左连接潜客小时宽表、当前截面和业财。虽然本批潜客宽表现已补齐，但未来首日查询若先从该表出发，仍可能在跨日补数前漏掉真实转移线索。
