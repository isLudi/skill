# Explore Data Skill 中文使用说明

## 这个 skill 是做什么的

`explore-data` skill 用于在正式分析或写看板 SQL 前，对数据集、数据库表、SQL 查询结果、CSV、Excel、Parquet、JSON 等数据做初步画像。它重点回答：这份数据有多少行列、粒度是什么、哪些字段适合做维度或指标、空值率如何、是否有重复、异常值、日期覆盖是否完整、哪些字段可能是 join key，以及通过 USQL RestAPI 实际能查哪些表、哪些部门或范围。

这个文件是中文上手说明，实际执行规则仍以同目录下的 `SKILL.md` 为准。

## 适合使用的场景

- 第一次接触一张新表，需要先了解字段、粒度、主键和数据质量。
- 写 SQL 前，想知道哪些字段适合做维度、指标、时间字段或 join key。
- 看板数据异常时，先检查源表是否有空值、重复、缺失分区、异常枚举值。
- 对 CSV、Excel、Parquet、JSON 文件做字段画像和数据质量初筛。
- 需要为一张表生成简要 schema 文档、字段分类和后续分析建议。
- 需要摸清 USQL RestAPI 权限边界：哪些表可查、哪些字段被拦截、部门或行级权限限定在哪里。
- 在正式使用 `validate-data` 审阅结论前，先做数据底层探查。

## 基本使用方法

在 Codex 中直接说明要使用 `explore-data` skill，并提供表名、文件路径、SQL 查询结果或样例数据即可。

推荐提供这些信息：

- 数据来源：完整表名、文件路径、SQL 结果或粘贴的数据。
- 探查范围：日期、`dt`、`hour`、期次、部门、项目或其他过滤条件。
- 业务目的：为了写 SQL、排查异常、找 join key、做指标分析，还是生成表画像。
- 重点关注字段：主键、部门字段、时间字段、金额/转化/排名/计数字段等。
- 是否允许通过 USQL RestAPI 对内部 Presto 表做小范围查询。
- 是否需要沉淀权限边界发现，用于后续完善知识库或申请更多表/部门权限。

## 常用 Prompt 模板

### 探查内部 Presto 表

```text
请使用 explore-data skill 探查这张表：
<完整库名.表名>

范围限定：
dt = '<YYYYMMDD>'
如果需要调用 USQL RestAPI，请只做只读查询，先抽样和小范围 profile，不要全表扫描。
请输出表粒度、候选主键、重要字段分类、空值率、top values、候选 join key 和后续分析建议。
```

### 检查一张表是否适合写看板 SQL

```text
请使用 explore-data skill 检查 <完整库名.表名> 是否适合用于看板取数。
重点关注：
1. 粒度是否明确；
2. 是否有 dt/hour 或其他分区字段；
3. 部门、期次、顾问、线索、订单等关键字段是否可用；
4. 是否存在重复或 join 膨胀风险；
5. 哪些字段适合作为维度和指标。
```

### 探查本地文件

```text
请使用 explore-data skill 探查这个文件：<文件路径>
请输出行列数、字段类型、空值率、重复行、异常值、日期范围、字段分类和推荐分析方向。
不要覆盖原文件。
```

### 排查数据质量问题

```text
请使用 explore-data skill 排查这份数据的质量问题。
现象是：<描述异常>
请重点检查空值、重复、异常枚举、日期缺口、负数/零值异常、主键唯一性和候选 join key。
```

### 探查 USQL 权限边界

```text
请使用 explore-data skill 探查以下表在 USQL RestAPI 下的权限边界：
<表名列表>

请按最小安全查询执行：
1. 先用 select 1 as smoke_test 验证接口连通；
2. 对每张表用 dt/日期/部门等安全范围做 limit 1 测试；
3. 区分表无权限、字段无权限、部门/行级范围受限、SQL 语法错误、空结果但可读；
4. 输出权限矩阵，并说明若要申请权限，需要申请哪些表、字段、部门或业务范围。

不要输出 token、账号、cookie 或 env 内容。
```

### 生成表画像文档

```text
请使用 explore-data skill 为 <完整库名.表名> 生成一份表画像文档。
内容包括：表用途、粒度、主键候选、核心字段、字段分类、典型过滤条件、常见 join key、已知数据质量问题和示例查询。
```

## 常见工作流

### 结构理解

先确认数据来源、范围条件和表粒度。对内部表，优先读取 `sql-query-writer-for-dashboard` 中已有的表文档和表索引；对文件，先读取表头和少量样例行。

### 安全探查

对内部 Presto 表，只执行只读 `SELECT`。大表必须限制 `dt`、`hour`、日期、部门或其他范围，不做无条件全表扫描。抽样查询必须加 `limit`。

### 权限边界探查

公司数据库权限可能同时包含接口权限、表权限、字段权限、部门/行级权限和查询平台解析限制。探查时要把它们拆开记录：

- 接口是否可用：`select 1 as smoke_test` 是否成功。
- 表是否可读：最小 `select 1 from 表 where 安全范围 limit 1` 是否成功。
- 字段是否可读：逐步增加字段，定位是否存在字段级拦截。
- 部门或范围是否可见：对已知部门、期次、业务线做小范围 count，区分“无权限”和“有权限但无数据”。
- 查询平台是否误拦截：记录权限解析器误判的 SQL 写法，避免后续重复踩坑。

输出时使用权限矩阵，记录表名、测试范围、结果、必须范围字段、错误特征和权限申请建议。

### 字段画像

对关键字段检查：

- 空值数和空值率。
- distinct 数和基数比例。
- top values 和异常枚举值。
- 数值字段的最小值、最大值、均值、分位数、零值和负值。
- 日期字段的最小/最大日期、未来日期、缺失分区或明显断点。

### 关系发现

识别可能的主键、外键、join key 和层级关系。例如顾问、部门、期次、线索、订单、渠道、班型等字段之间是否存在一对一、一对多或多对多风险。

### 输出建议

最后给出哪些字段适合做维度、哪些字段适合做指标、哪些时间字段适合做趋势分析，以及下一步应该跑哪些 SQL 或做哪些校验。

## 和本地 SQL skill 的配合方式

`explore-data` 适合作为 `sql-query-writer-for-dashboard` 的前置探查工具：

- 先用 `explore-data` 了解表结构、字段质量、粒度和 join key。
- 再用 `sql-query-writer-for-dashboard` 生成符合公司规则的 Presto SQL。
- 查询结果出来后，再用 `validate-data` 做交付前审阅和风险检查。

如果任务是“写生产可用 SQL”，优先使用 `sql-query-writer-for-dashboard`；如果任务是“先看看这张表/文件长什么样”，优先使用 `explore-data`。

## 依赖和环境

根据任务不同，可能使用：

- `D:\anaconda3\python.exe`：本机 Codex 默认 Python 解释器。
- `pandas`：读取和分析 CSV、Excel、Parquet、JSON 等数据。
- `openpyxl`、`python-calamine`：读取 Excel 文件。
- `sql-query-writer-for-dashboard` 中的 USQL RestAPI 规则：对内部 Presto 表做小范围只读查询。

## 使用注意

- 对内部表探查时不要输出或保存真实 token、账号、cookie、env 内容。
- 对大表不要做无范围全表扫描；优先按日期、分区、部门或期次缩小范围。
- 权限探查不要盲扫全库或暴力枚举部门；只探查用户给定、知识库已有或用户明确批准的候选表和范围。
- 表“可查询”不等于“所有部门都可见”；必须单独记录部门、业务线、期次或行级范围边界。
- 如果只用了抽样数据，结论必须标注为“基于样本”。
- 如果只有表结构没有实际数据，不要编造空值率、分布或 top values。
- 数据质量告警是启发式判断，不一定都是错误，但每个高风险项都应进一步核实。
